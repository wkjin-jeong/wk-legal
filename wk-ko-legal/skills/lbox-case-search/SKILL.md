# lbox-case-search

lbox.kr 판례 검색 사이트에서 사용자 질의와 관련된 판례를 자동으로 검색하고, 본문을 조회하여 사용자 질의에 답하는 markdown 보고서를 생성합니다.

> **2026.6 사이트 전면 개편 반영본.** lbox.kr은 "법률가를 위한 AI 업무 환경"으로 개편되어, 판례 검색이 **별도 검색 URL**이 아니라 홈 화면 **컴포저(composer)의 "검색" 모드 → 작업(Task)** 흐름으로 바뀌었습니다. 본문 페이지 URL도 `/v2/case/...` → `/precedent/...`로 변경되었습니다. 개편 전 방식(직접 검색 URL, `<script>` 페이로드 복원)은 더 이상 동작하지 않으므로 사용하지 마세요. 핵심 변화 요약은 이 문서 끝의 "개편 전 대비 변경점"을 참고하세요.

## 사전 조건 및 환경 점검

작업 시작 전에 빠르게 점검하세요.

1. **Chrome 실행 + lbox.kr 로그인 필수**: lbox.kr은 로그인하지 않으면 검색·본문에 접근할 수 없습니다(판례 본문 API가 401을 반환). 사용자가 Chrome에서 미리 로그인해 두었다는 전제로 동작합니다. 이 Skill은 Chrome 확장(Claude in Chrome)의 **계정 기반 연결**로 작동하므로 Claude가 동작하는 PC와 Chrome 브라우저가 동작하는 PC는 **같을 필요가 없습니다**. 단 lbox.kr 로그인은 Chrome이 실제 실행 중인 PC에서 사용자가 직접 마쳐 두어야 합니다.

2. **Chrome MCP 도구 선택**: `mcp__Claude_in_Chrome__*`(`navigate`, `javascript_tool`, `read_page`, `computer`, `read_network_requests`, `tabs_context_mcp` 등)를 사용합니다. JS 실행과 DOM 추출이 안정적이므로 이 도구를 사용하세요. 컴포저에 검색어를 입력하고 모드를 선택하는 단계에는 `computer`(클릭·타이핑) 또는 `find`/`form_input`이 필요합니다.

3. **연결된 브라우저 확인 + 선택**:
   - `list_connected_browsers`로 같은 Anthropic 계정에 연결된 Chrome 확장 인스턴스 목록을 확인합니다(`deviceId`, `name`, `osPlatform`, `isLocal`).
   - 여러 기기가 연결되어 있으면 어느 기기에서 lbox.kr에 로그인했는지 사용자에게 확인한 뒤 `select_browser(deviceId)`로 작업 대상 브라우저를 지정합니다.
   - 각 대화는 `tabs_context_mcp{createIfEmpty:true}`로 전용 탭을 확보한 뒤 그 `tabId`로 작업합니다.

4. **검색은 '작업(Task)'을 생성합니다 (동작 안내)**: 개편된 lbox에서 "검색" 모드로 질의하면 사용자 계정의 **작업(Task) 목록에 항목이 하나 생성**됩니다(예: 좌측 메뉴 "작업", URL `/task/{id}`). 이는 새 lbox에서 검색의 정상 동작이며, 사용자의 **등록 사건(lbox-case-progress가 다루는 '사건')과는 별개**입니다. 한 번의 검색 세션에서는 가급적 **하나의 작업만** 만들고 그 안에서 페이지 순회·재검색을 진행하세요. 사용자가 원치 않으면 작업은 사용자가 직접 삭제할 수 있습니다.

5. **JS 실행 사전 점검**: 첫 페이지에서 `document.title` 등을 평가해 정상 응답이 오는지 확인합니다. 실패하면 사용자에게 확장 권한·연결 상태 점검을 요청하고 중단하세요.

## 작업 흐름

### 1단계: 사용자 의도 파악

사용자 질의에서 다음을 추출합니다.

- **검색어 (필수)**: 핵심 법률 쟁점/키워드. 자연어로 길게 물어봐도 lbox 검색에 적합한 짧은 키워드 형태로 선정합니다. (예: "회사가 갑자기 해고했는데 부당해고로 인정될 수 있나요?" → "부당해고")
- **사건유형 (선택)**: 민사/형사/행정/가사. 새 UI에서는 검색 결과 패널의 **사건유형 필터**로 적용합니다.
- **법원 (선택)**: 사용자가 '대법원'을 명시하면 결과 패널의 **법원 필터**로 한정합니다.
- **선고기간 (선택)**: '최근 1년/3년/5년'은 결과 패널의 **선고일 필터**로 적용합니다.

선택 파라미터를 적용하는 가장 간단한 방법은 **검색어 자체에 자연어로 녹이거나**(예: "대법원 부당해고"), 결과 패널의 필터 드롭다운을 사용하는 것입니다(필터 조작은 `references/extraction.md` 1장 참고). 명시되지 않은 선택 파라미터는 적용하지 않습니다.

질의가 모호하거나 검색어 선정이 불확실하면 진행 전에 한 가지만 사용자에게 확인하세요.

### 2단계: 검색 실행 (컴포저 "검색" 모드)

개편된 lbox에는 별도 검색 URL이 없습니다. 홈 화면 컴포저에서 검색합니다.

1. `navigate`로 `https://lbox.kr/` 로 이동합니다.
2. 컴포저 입력창(placeholder "내용을 입력하세요")에 검색어를 입력하고, 모드 아이콘에서 **"검색"(돋보기)** 을 선택한 뒤 **제출(↑)** 합니다.
3. 제출하면 `/task/{id}` 작업 페이지로 이동하고, **검색 결과 패널**이 열립니다(판례 탭 기준 "○○ 검색결과 N개").

구체적 클릭·타이핑 좌표 잡는 법, 모드 선택 확인, 결과 패널이 닫혀 있을 때 "검색 결과" 카드를 클릭해 다시 여는 법은 **`references/extraction.md` 1장을 이 시점에 읽고 따릅니다.**

> **검색 후 짧게 대기**: `navigate`·제출 직후 곧바로 JS를 실행하면 렌더링과 충돌해 `Runtime.evaluate timed out`이 날 수 있습니다. 제출(클릭) 호출과 결과 추출 `javascript_tool` 호출을 **분리**하고 사이에 짧은 텀을 두세요.

### 3단계: 결과 카드 추출 및 페이지 순회

결과 패널의 **판례 탭**에서 카드를 추출합니다. 각 카드는 `a[data-track-props]`(JSON에 `documentType:"precedent"`)이며, `data-track-props.docId`(`"법원-사건번호"` 형식), `rank`, 그리고 카드 텍스트(법원·선고일·사건번호·사건명·스니펫·**주문 결과 배지**(파기환송/상고기각/원고일부승 등)·**인용 수**·**조회 수**)를 제공합니다. 추출 JS와 `docId` → 본문 URL 변환 규칙은 `references/extraction.md` 1장을 따릅니다.

#### 페이지 순회 정책

- 결과는 **10건/페이지**, 하단에 **번호 페이지네이션(1 2 3 4 5 … »)** 이 있습니다. 결과는 기본 **관련도순** 정렬입니다.
- **항상 최소 3페이지(상위 30건)까지** 살펴봅니다. 상위 결과일수록 관련도가 높으므로, 개편 전의 5페이지 정책보다 적은 페이지로도 동등 이상 품질을 확보할 수 있습니다.
- 3페이지까지 누적된 결과 중 **High 관련성 판례가 10건 이상**이면 거기서 멈춥니다.
- 3페이지까지 끝났는데도 High가 10건 미만이면, 4페이지부터 한 페이지씩 더 진행하여 (누적 High ≥ 10) 또는 (마지막 페이지 도달)에서 멈춥니다.
- 페이지 이동은 하단 번호 버튼을 클릭합니다(스크롤하면 페이지네이션이 보입니다). `references/extraction.md` 1장 참고.

#### 스니펫 1차 관련성 판단 (snippet triage)

본문 visit이 비싸므로, 우선 **카드의 스니펫·결과 배지·인용 수만 보고 1차 관련성**을 매깁니다.

- **High 후보**: 사용자 질의의 핵심 쟁점이 스니펫에 직접 드러난 카드. **인용 수가 많은 대법원 판례**는 그 법리의 안정성·선례성이 높음을 시사하므로 우선 순위를 높입니다.
- **Medium 후보**: 검색어는 일치하지만 쟁점이 약간 다르거나 스니펫만으로 판단이 어려운 카드.
- **Low**: 검색어만 우연히 맞고 쟁점이 명확히 다른 카드. 본문 visit을 건너뛰고 사건번호만 별도 목록으로 모아둡니다("정리에서 제외한 판례"에 사용).

본문은 **High + Medium 후보만** visit합니다.

### 4단계: 각 판례 본문 조회

`docId`(`"법원-사건번호"`)를 `https://lbox.kr/case/{법원}/{사건번호}`(각 부분 `encodeURIComponent`)로 합쳐 `navigate`합니다(구 `/precedent/…` 경로도 `/case/…`로 자동 리다이렉트되어 작동).

> **본문은 이제 DOM에서 바로 읽힙니다 (개편의 가장 큰 단순화).** 개편 전의 가상화·비활성 탭 문제가 해소되어, 탭이 `hidden`이어도 `document.body.innerText`와 단락 DOM이 정상적으로 채워집니다(전원합의체 장문 판결도 전 단락 적재 확인). **개편 전의 `<script>` 페이로드 복원 방식은 폐기**합니다 — 이제는 페이지에 섞인 다른 자료(최근 본 자료·추천 판례) 텍스트까지 끌어와 노이즈만 늘립니다.

각 판례 페이지에서 다음을 추출합니다(셀렉터·JS는 `references/extraction.md` 2장).

- **법원·판결 종류**: `data-node-id="lbox-paragraph-topheader-1"`(법원), `topheader-2`(판결/결정). 제목 전체는 `document.title`("대법원 2013. 12. 18. 선고 2012다89399 전원합의체 판결 [퇴직금] - LBOX 판례")에서도 회수.
- **사건정보(사건번호·당사자·원심판결)**: `lbox-paragraph-before-*` (특히 `before-1`이 전체 블록).
- **판시사항**: `lbox-paragraph-issue-*`.
- **판결요지**: `lbox-paragraph-summary-*`.
- **본문(주문·이유·의견)**: `lbox-paragraph-main-*` (번호순 정렬). 전원합의체는 다수의견·반대의견·별개의견·보충의견 포함.
- **재판부**: `lbox-paragraph-judges-*`.

**적재(완전성) 판정**: `main` 단락 수가 0이 아니고, 마지막 `main`이 결론부("…주문과 같이 판결한다", "보충의견을 밝힌다" 등)로 끝나거나 `judges`가 존재하면 정상입니다. 본문이 비고 URL이 로그인 페이지로 리다이렉트된 경우에만 재로그인을 의심하세요(특정 한 건만 비면 건너뛰고 진행 상황을 한 줄로 알림).

### 4.5단계: 관련 판결 추적 (필수)

판례 본문 페이지 **우측 사이드바**에 관련 자료가 구조화되어 있습니다. 이 정보가 답변 품질에 결정적입니다.

- **상·하위 판결(소송 진행 체인)**: 사이드바 "상·하위 판결" 섹션. 각 항목은 `[data-track-click="upperLowerCaseItem"]`이며 `data-track-props.docId`와 **관계 라벨**(원고승/원고패/파기환송/상고기각/원고일부승 등) + 법원 + 사건번호를 텍스트로 제공합니다. 1심→2심→상고심 흐름이 그대로 나옵니다.
- **인용된 판례 / 따름 판례**: "인용된 판례 N", "따름 판례 N" 섹션(접힘). 핵심 대법원 선례·따름 판례가 많은 선례는 펼쳐서 사건번호를 확보합니다.

##### (가) 상급심 판결의 의무적 조회 — 하급심 본문을 visit한 경우 반드시 적용

**대법원이 아닌 하급심 판례의 본문을 visit한 경우, 사이드바 "상·하위 판결"에 그 사건의 상급심(항소심·상고심)이 있으면 그 상급심 본문도 반드시 함께 `navigate`해서 조회합니다.** 예외 없는 필수 절차입니다. 이유:

- 같은 사실관계에 대해 상급심이 결론을 뒤집을 수 있습니다(특히 **파기환송**).
- 항소·상고 기각이라도 상급심이 결론에 이르는 이론을 다르게 설시할 수 있고, 그 부분이 답변에 결정적일 수 있습니다.
- 하급심만 보고 답변을 쓰면 **잘못된 결론**을 전달할 위험이 큽니다.

조회 절차: ① "상·하위 판결" 체인에서 상급심 `docId`를 확인 → `/case/{법원}/{사건번호}`로 navigate, ② 1심→2심→대법원이 다단계면 **가장 상급심까지** 추적, ③ 각 상급심의 주문·핵심 이유를 확인해 하급심과 어긋나는지 점검, ④ 상급심이 뒤집은 경우 답변에는 **상급심 결론을 우선** 사용하고 하급심은 "원심/환송 전 판결"로 위치지어 인용("결론" 항목에 "원고일부승(1심) → 파기환송(대법원) → 항소기각(환송 후 2심)" 식으로 흐름 명시).

셀렉터·추출 JS는 `references/extraction.md` 3장을 따릅니다.

##### (나) 참조·따름 판례의 추적

- 핵심 대법원 선례가 사이드바 "인용된 판례"에 있으면 답변의 "참조 대법원 선례 정리" 표에 기록하고, 반복 인용되는 선례는 본문도 visit해 정확한 판시사항·판결요지를 확보합니다.
- "따름 판례" 수가 많은 대법원 선례는 법리 안정성이 높으므로 답변에서 우선 인용합니다.

### 5단계: 관련성 필터링 (강화)

본문 visit이 끝나면 각 판례를 **본문 기반 최종 관련성**으로 재분류합니다. 스니펫에서 High였더라도 본문 확인 결과 쟁점이 달랐다면 강등하고, Medium이었는데 본문이 쟁점을 명확히 다뤘다면 승격합니다.

- **직접 관련(High)**: 질의 쟁점을 정면으로 다룬 판례 → "참고 판례"에 사건명·판시사항·판결요지·관련 부분 모두 포함.
- **간접 관련(Medium)**: 검색어는 일치하나 쟁점·맥락이 약간 다른 판례 → 비교 가치가 있을 때만 한 줄 요지로 언급.
- **관련성 낮음(Low)**: 검색어만 우연히 일치 → "정리된 답변"·"참고 판례"에서 제외하고, 별도 섹션에 사건번호만 한 줄로 모아둡니다.

이 Skill의 가치는 링크 나열이 아니라 **사용자 질문에 답하는 것**입니다. 노이즈 판례를 끼워 넣으면 답변 품질이 떨어집니다.

### 5.5단계: 직접 사례 부족 시 재검색 정책

본문 visit과 관련 판결 추적이 끝난 시점에 질의의 **정확한 사실관계/쟁점**을 다루는 "직접 사례"가 충분한지 점검합니다. 다음 중 하나면 "직접 사례 부족"으로 판정합니다.

- 핵심 쟁점을 정면으로 다룬 판례가 1건 이하.
- 발견된 선례의 사실관계가 동떨어져 다른 분야 법리를 유추 적용해야만 하는 경우.
- High 다수가 "검색어 일치"일 뿐 "쟁점 일치"가 아닌 경우.

부족하면 키워드를 바꿔 재검색합니다(2~4단계 재실행). 같은 작업 안에서 컴포저에 새 검색어를 다시 제출하면 됩니다.

- **1차 재시도**: 좁은↔넓은 키워드 전환, 동의어·상위어·하위어(예: 부당이득↔원상회복, 해제↔해지·취소, 통지↔대항요건), 사건유형·법원 필터 추가.
- **2차 재시도**: 1차와 다른 방향으로 변경.
- 진행 전 사용자에게 한 줄로 알립니다: "직접 사례가 부족하여 `<사유>`를 위해 `<새 키워드>`로 재검색합니다." 결과는 기존과 **합산**하여 하나의 markdown으로 정리하고, "검색 조건"에 사용한 모든 검색어를 기록합니다.
- 2차까지도 부족하면 답변을 작성하되, **첫머리와 말미 두 곳**에 한계점을 명시합니다(형식은 `references/report-format.md`).

### 6단계: 결과 markdown 파일 작성

**저장 위치**: 사용자가 선택한 작업 폴더에 저장합니다. 없으면 outputs/에 저장하고 `computer://` 링크로 안내합니다. 절대 경로를 하드코딩하지 마세요.

**파일명**: `lbox-search-<주요키워드>-<YYYYMMDD>.md` (예: `lbox-search-부당해고-20260629.md`). 민감정보(실명·주민등록번호·계좌번호 등)는 파일명에 넣지 않습니다.

보고서 표준 양식(markdown 템플릿, 결론을 가르는 사실관계 변수·상급심 흐름 표기, 직접 사례 부족 시 첫머리·말미 **두 곳** 한계 표기, `computer://` 안내)은 **`references/report-format.md`를 작성 직전에 읽고 그대로 따릅니다.**

## 인용/출처 작성 원칙

- **모든 사실 진술에는 출처(사건번호)가 있어야 합니다.** 사용자가 원문을 추적할 수 있어야 합니다.
- **사건번호 표기**는 가능하면 법원명을 함께 적습니다(`대법원 2023다12345`).
- 본문에 없는 내용을 추정해서 적지 마세요. 확인되지 않은 내용은 "검색된 판례에서는 명확히 확인되지 않음"으로 표시합니다.
- 원문 링크는 `https://lbox.kr/case/{법원}/{사건번호}` 형식으로 적습니다.

## 문제 발생 시

증상별 대응(검색 0건, `Runtime.evaluate timed out`, `[BLOCKED]`, truncated, `Tab not found`, 로그인 리다이렉트, 결과 패널이 닫힘, 셀렉터 불일치 등)은 `references/troubleshooting.md` 참조. 핵심: ① 본문은 이제 DOM에서 바로 읽히므로 `<script>` 복원을 쓰지 말 것, ② `navigate`/제출과 `javascript_tool`은 분리 호출하고 짧은 텀을 둘 것(JS 내부에 대기 루프 금지), ③ 검색 결과가 안 보이면 timeline 카드("○○ 검색 결과")를 클릭해 패널을 다시 연다.

## 금지 사항

- 사용자가 lbox에 로그인하지 않은 상태에서 강제로 진행하지 마세요.
- 본문에서 확인되지 않은 법리를 지어내지 마세요.
- 판례를 단순 나열하지 말고 **사용자 질의에 답하는 형태**로 정리하세요.
- 검색어만 우연히 일치하고 쟁점이 다른 판례를 그대로 끼워 넣어 노이즈를 만들지 마세요.
- **하급심 본문을 visit한 뒤 사이드바 "상·하위 판결"에 상급심이 있는데도 그 상급심 본문을 조회하지 않은 채 답변을 작성하는 것은 금지**됩니다.
- **직접 사례가 부족한데도 1차·2차 재검색을 생략한 채 곧바로 답변을 작성하는 것은 금지**됩니다.
- 개편 전의 폐기된 방식(직접 검색 URL `/v2/search/case`, 본문 `<script>` 페이로드 복원)을 사용하지 마세요.
