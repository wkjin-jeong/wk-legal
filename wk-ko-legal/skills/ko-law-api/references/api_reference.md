# 국가법령정보 OPEN API 참조 (skill 내부 문서)

본 문서는 `scripts/law_api.py`가 호출하는 `law.go.kr/DRF` API의 파라미터·응답 필드·인코딩 규칙을 요약한다.
공식 가이드: https://open.law.go.kr/LSO/openApi/guideList.do

---

## 1. 베이스 URL과 target 매트릭스

| 베이스 URL | 용도 |
|---|---|
| `https://www.law.go.kr/DRF/lawSearch.do` | 목록 검색 |
| `https://www.law.go.kr/DRF/lawService.do` | 본문 조회 |

| target | 대상 | 검색 결과의 식별자 필드 | 본문 조회 시 보낼 파라미터 |
|---|---|---|---|
| `law` | 법령(법률·시행령·시행규칙·대통령령 등) | `법령일련번호` (=MST) / `법령ID`(6자리) | **`MST=<법령일련번호>`** |
| `admrul` | 행정규칙(고시·훈령·예규·감독규정·지침) | `행정규칙일련번호` (긴 숫자, 약 13자리) / `행정규칙ID`(짧은 숫자) | **`ID=<행정규칙일련번호>`** |
| `ordin` | 자치법규(조례·규칙) | `자치법규일련번호` (=MST) / `자치법규ID` | **`MST=<자치법규일련번호>`** ⚠ |
| `licbyl` | 법령 별표·서식 | (검색 결과의 다운로드 URL 사용) | 검색 전용 — 본문 API 없음 |
| `admbyl` | 행정규칙 별표·서식 | (검색 결과의 다운로드 URL 사용) | 검색 전용 — 본문 API 없음 |
| `ordinbyl` | 자치법규 별표·서식 | (검색 결과의 다운로드 URL 사용) | 검색 전용 — 본문 API 없음 |
| `expc` | 법령해석례 (법제처 유권해석) | `법령해석례일련번호` (예: 332741) | **`ID=<법령해석례일련번호>`** |
| `eflaw` | 현행법령(시행일) — 연혁 포함 버전 | `법령일련번호`(버전별 MST) / `법령ID` | **`MST=` + `efYd=`(필수)** ⚠ `ID=`는 현행 반환 |
| `admrulOldAndNew` | 행정규칙 신구법비교 | `행정규칙일련번호` | **`ID=<행정규칙일련번호>`** (또는 `LID=<행정규칙ID>`) |

> **자치법규(`ordin`) 본문 호출은 `MST=` 파라미터를 사용**한다(`ID=`가 아님).
> 검색 결과의 `<자치법규상세링크>`가 `target=ordin&MST=...` 형태로 나오는 것에서 확인됨.

> `MST`(마스터번호)는 한 법령·자치법규의 **시점별 본문**을 가리키는 식별자다.
> 동일 법령이라도 개정 이력이 있으면 시점별로 서로 다른 MST가 부여된다.
> 시행 중인 본문을 인용하려면 검색 결과에서 **시행일자 ≤ 오늘인 항목 중 가장 최신**을 고른다.
> 검색 결과의 `<현행연혁코드>` 값이 `현행`인 항목이 우선 후보다.

> 행정규칙 본문 조회 시 헷갈리지 말 것: `<행정규칙ID>`(예: 21828)가 아니라
> `<행정규칙일련번호>`(예: 2100000274812)를 `ID=` 파라미터에 넣어야 한다.

## 2. 공통 파라미터

| 파라미터 | 필수 | 의미 |
|---|---|---|
| `OC` | ✅ | 인증키(이메일 ID 형식, open.law.go.kr 신청). 환경변수 `LAW_GO_KR_OC`로 관리 권장. |
| `target` | ✅ | `law` / `admrul` / `ordin` 등 |
| `type` | ✅ | 응답 포맷: `XML`(권장) / `JSON` / `HTML` |

## 3. 검색 API (`lawSearch.do`) 파라미터

| 파라미터 | 의미 | 비고 |
|---|---|---|
| `query` | 검색어 | 법령명·규칙명·조례명 등 |
| `display` | 결과 개수 (최대 100) | target별 기본값: `law`/`admrul`/`expc`=20, `ordin`/`licbyl`/`admbyl`/`ordinbyl`=50 (분야 키워드로 결과가 흔히 수백~수만 건이라 첫 페이지를 넓게 봐야 함) |
| `page` | 페이지 번호 | 1부터 시작 |
| `search` | 검색범위 | `1`=법령명(기본), `2`=본문 |
| `org` | 소관부처 | 행정규칙·자치법규에서 유용 |
| `knd` | 종류 코드 | target별 의미 다름 |
| `efYd` | 시행일자 범위 | `YYYYMMDD~YYYYMMDD` |
| `ancYd` | 공포일자 범위 | `YYYYMMDD~YYYYMMDD` |
| `sort` | 정렬 | `lasc`/`ldes`(법령명), `dasc`/`ddes`(공포일), `ndes` 등 |

## 4. 본문 조회 API (`lawService.do`) 파라미터

| 파라미터 | 의미 | 비고 |
|---|---|---|
| `MST` | 법령 마스터번호 | target=law에서 가장 권장 |
| `ID` | 법령ID/행정규칙ID/자치법규ID | target에 따라 의미 다름 |
| `LID` | 일부 행정규칙·자치법규에서 사용 | 발령번호 기반 식별자 |
| `LM` | 법령명·규칙명 직접 지정 | 검색 없이 단발 호출 시 |
| `JO` | 조문번호 | 6자리 zero-padding (예: 제2조 → `000200`, 제390조 → `039000`) |
| `efYd` | 시행일자 | `YYYYMMDD` |
| `ancYd` | 공포일자 | `YYYYMMDD` |
| `LANG` | 언어 | `KO`(기본), `EN`(영문번역본 — 일부 법령만 제공) |

## 5. 조문번호(`JO`) 인코딩 규칙

API는 조문번호를 **6자리 zero-padding 숫자**로 받는다.
일반적인 패턴은 다음과 같다.

| 표시 | JO 값 | 설명 |
|---|---|---|
| 제2조 | `000200` | 앞 4자리=조 번호, 뒤 2자리=가지조(없으면 00) |
| 제100조 | `010000` | |
| 제390조 | `039000` | |
| 제390조의2 | `039002` | 뒤 2자리에 가지조 번호 부여 |

`law_api.py`의 `--jo` 인자는 사용자가 다음 형태 중 어떤 것으로 입력해도 받아들이도록 처리한다.

- `390` (숫자만) → `039000`으로 자동 패딩
- `039000` (이미 6자리 형태) → 그대로 통과
- `제390조` (자연어) → 숫자 추출 후 패딩

> **가지조** 또는 **항·호 단위 인코딩**(예: `03900100` = 제390조 제1항)은 자릿수가 늘어난다.
> 정확히 의도한 경우에만 사용자가 직접 그 값을 `--jo`로 명시한다(스크립트가 임의로 추측하지 않음).

## 6. 자주 마주치는 응답 패턴과 대처

### (1) 검색 결과 0건
- 띄어쓰기·괄호·약칭을 바꿔가며 재시도.
- 행정규칙은 `--org`(소관부처)를 함께 지정하면 적중률이 올라간다.

### (2) 응답이 HTML(오류 페이지)로 옴
- OC 인증키가 잘못되었거나 미발급 상태.
- 일일 호출 한도 초과.
- 파라미터 조합이 잘못됨(필수 키 누락 등).

### (3) 본문이 `JO` 단위로 안 잘림
- 일부 응답은 전체 본문만 제공된다. 전체를 받은 뒤 **호출자 측에서 해당 조문만 발췌**한다.

### (4) 시행 전 법령(미시행본)이 검색됨
- 검색 결과에 `시행일자`가 미래인 항목이 함께 나올 수 있다.
- 인용 시점 기준의 **현행** 본문을 인용하려면 `시행일자 ≤ 오늘`인 항목 중 가장 최근 것을 사용한다.
- 시행 예정 개정 본문을 인용해야 한다면 그 사실을 서면에 명시한다("○년 ○월 ○일 시행 예정 개정 ○○법").

## 6-A. 별표·서식 (licbyl / admbyl / ordinbyl)

별표(別表)·서식(書式)은 법령·행정규칙·자치법규의 본문 끝에 붙는 첨부 파일성 자료다.
일반적으로 PDF·HWP/HWPX·이미지 형태로 제공되며, 본문 조회 API는 별도로 제공되지 않고
**검색 응답 안의 다운로드 URL을 따라가는 방식**으로 가져온다.

### 검색 URL

| target | 검색 엔드포인트 | section(검색 범위) | 응답 루트 | 항목 element |
|---|---|---|---|---|
| `licbyl` | `https://www.law.go.kr/DRF/lawSearch.do?target=licbyl&...` | `bylNm` (별표명) | `<licBylSearch>` | `<licbyl>` |
| `admbyl` | `https://www.law.go.kr/DRF/lawSearch.do?target=admbyl&...` | `admBylNm` (행정규칙 별표명) | `<admRulBylSearch>` | `<admrulbyl>` ⚠ |
| `ordinbyl` | `https://www.law.go.kr/DRF/lawSearch.do?target=ordinbyl&...` | `ordinBylNm` (자치법규 별표명) | `<licBylSearch>` ⚠ (licbyl과 동일!) | `<ordinbyl>` |

검색 파라미터는 다른 target과 동일(`OC`, `query`, `display`, `page`, `type` 등).

### ⚠ 검색 동작에 관한 핵심 주의 (라이브 검증으로 발견)

**별표·서식 검색의 `query`는 "별표 자체의 이름"을 본다.** 모법(법령·행정규칙·자치법규)의 이름이 아니다.
- "건축법" 같은 모법명으로 검색하면 0건 (별표명에 "건축법"이라는 글자가 직접 들어간 경우만 매칭).
- 별표 자체의 통칭(예: "안전점검 검사 결과", "위탁지정신청서") 또는 분야 키워드("옥외광고물") 단위로 검색.
- 모법별 별표를 모으는 흐름은 **(a) 모법(law/admrul/ordin)을 먼저 검색·본문 조회 → 본문 응답의 `<별표>` 영역을 보거나**, **(b) 본문 응답에서 별표 다운로드 URL이 함께 노출되는지 확인**해야 한다.

### `query=*` 와일드카드 (전체 목록 받기)

라이브 검증에서 `query=*`로 전체 별표·서식 목록을 받을 수 있음이 확인됨.
licbyl `*` → totalCnt=39,465 / admbyl `*` → totalCnt=83,249 (페이지네이션 필수).

```bash
python scripts/law_api.py search --target licbyl --query "*" --display 100 --page 1 --pretty
```

### ⚠ admbyl 응답의 특수 사항

행정규칙 별표·서식(admbyl)에는 다음과 같은 특이 사항이 있다.
- **`<별표명>`이 비어 있는 경우가 다수** (특히 대법원 가족관계등록예규·등기예규 등). 파일명이나 인용 시 `관련행정규칙명_별표<별표번호>` 형태로 폴백 명명한다(스크립트 `download` 명령은 자동 처리).
- **다운로드 링크(`<별표서식파일링크>`)가 없는 항목 존재**. 이 경우 별표가 본문에 포함된 형태이므로 별도 다운로드가 불가능. 스크립트는 해당 항목을 스킵하고 카운트만 보고한다.
- **`<별표서식PDF파일링크>`가 제공되지 않음** (licbyl에만 PDF 변환본 제공).

### 응답에서 다운로드 URL 식별 (우선순위)

응답 XML의 항목별 자식 태그 중 다음을 다운로드 링크로 사용한다 — **PDF 변환본 우선**.

| 우선순위 | 태그명 | 비고 |
|---|---|---|
| 1 (최우선) | `<별표서식PDF파일링크>` | licbyl에서만 제공. PDF로 변환된 사본 — 텍스트 추출에 가장 유리 |
| 2 | `<별표서식파일링크>` | 모든 byl target에서 제공. 원본 형식(보통 HWP, 일부 PDF·이미지) |
| 후보(보수적) | `<별표파일링크>`, `<별지파일링크>`, `<서식파일링크>` | 응답 변형 대비 |

스크립트의 `download` 명령은 위 우선순위를 자동 적용하며, `is_pdf` 플래그로 PDF 여부를 표시한다.
상대경로(`/LSW/...`, `/DRF/...`)는 자동으로 호스트(`https://www.law.go.kr`)를 붙여 절대화한다.
실제 응답에서 위에 없는 새 태그명을 발견하면 `BYL_LINK_TAG_HINTS_PDF` 또는 `BYL_LINK_TAG_HINTS_RAW`(scripts/law_api.py)에 추가한다.

### 별표 단건 상세 조회 (보너스)

응답에 포함된 `<별표자치법규상세링크>`(또는 유사 필드)는 다음 형식이다.
```
/DRF/lawService.do?OC=<oc>&target=ordinbyl&ID=<별표일련번호>&type=HTML
```
즉 별표 자체에 대한 상세 페이지 조회 API도 존재(target은 검색과 동일하지만 `lawService.do`에 `ID=<별표일련번호>`를 보냄).
필요 시 `--type HTML`로 호출하여 별표 본문 HTML을 받을 수 있다.

### 다운로드·텍스트 추출 흐름

```bash
# 1) 별표서식 목록 검색 + 응답 저장
python scripts/law_api.py search --target licbyl --query "건축법" \
    --display 20 --pretty --save-to ./byl_search_licbyl.xml

# 2) 검색 응답에서 추출된 모든 다운로드 URL을 일괄 다운로드 (PDF는 텍스트 추출까지)
python scripts/law_api.py download --from-search-xml ./byl_search_licbyl.xml \
    --extract-text --limit 5

# 3) 단건 직접 다운로드
python scripts/law_api.py download --url "/DRF/...별표파일URL..." \
    --filename "건축법_별표1" --extract-text
```

> **PDF 텍스트 추출**은 시스템에 `pdftotext`(poppler-utils)가 설치되어 있어야 한다.
> macOS: `brew install poppler` / Ubuntu: `apt install poppler-utils`.
> HWP/HWPX는 자동 텍스트 추출 미지원 — 파일만 저장하고 사용자에게 안내한다.

### 별표·서식 인용 시 유의

- 별표는 형식상 본문의 일부지만, 인용 시 **「○○법」 [별표 ○]** 표기를 사용한다.
- 서식은 **「○○법 시행규칙」 [별지 제○호서식]** 처럼 표기.
- 별표를 본문에 옮길 때는 표 구조가 깨지지 않게 가능한 한 PDF에서 추출한 텍스트 그대로 옮기고,
  표 형태가 중요하면 PDF 페이지 번호를 함께 명시("별표 1, 2면 표")한다.

## 6-B. 법령해석례 (expc)

법제처가 정부기관·지자체의 질의에 대해 회신한 **유권해석**. 법률 자문의견서 작성 시 권위 있는 보강 근거가 된다.

### 검색·본문 URL

| 용도 | 엔드포인트 | 응답 루트 |
|---|---|---|
| 목록 검색 | `https://www.law.go.kr/DRF/lawSearch.do?target=expc&...` | `<Expc>` |
| 본문 조회 | `https://www.law.go.kr/DRF/lawService.do?target=expc&ID=...` | `<ExpcService>` |

검색 `section`은 `itmNm` (안건명 기준).

### 검색 응답 구조 (라이브 검증)
```xml
<Expc>
  <target>expc</target>
  <키워드>임차</키워드>
  <section>itmNm</section>
  <totalCnt>57</totalCnt>
  <expc id="1">
    <법령해석례일련번호>332741</법령해석례일련번호>           <!-- 본문 조회 시 ID= -->
    <안건명><![CDATA[ 감사원 - 외국회사 소유의 항공기를 ... ]]></안건명>
    <안건번호>21-0624</안건번호>                              <!-- 법제처 안건번호 (YY-NNNN) -->
    <질의기관명>감사원</질의기관명>
    <질의기관코드>1040000</질의기관코드>
    <회신기관명>법제처</회신기관명>
    <회신기관코드>1170000</회신기관코드>
    <회신일자>2021.12.07</회신일자>                           <!-- ⚠ YYYY.MM.DD 형식 (다른 target은 YYYYMMDD) -->
    <법령해석례상세링크>/DRF/lawService.do?OC=...&target=expc&ID=332741&type=HTML</법령해석례상세링크>

```

### 본문 응답 구조 (라이브 검증)
```xml
<ExpcService>
  <법령해석례일련번호>315191</법령해석례일련번호>
  <안건명><![CDATA[ 여성가족부 - 건강가정기본법 제35조 제2항 관련 ]]></안건명>
  <안건번호>15-0105</안건번호>
  <해석일자>20150327</해석일자>                              <!-- ⚠ YYYYMMDD 형식 (검색의 회신일자와 다름) -->
  <해석기관코드>1170000</해석기관코드>
  <해석기관명>법제처</해석기관명>
  <질의기관코드>1383000</질의기관코드>
  <질의기관명>여성가족부</질의기관명>
  <관리기관코드/>
  <등록일시>20180109</등록일시>

  <!-- 본문 핵심 3단 구조 — 자문의견서 인용에 그대로 활용 -->
  <질의요지><![CDATA[ ... 질의의 배경과 내용 ... ]]></질의요지>
  <회답><![CDATA[ ... 결론 ... ]]></회답>

```

### 활용 흐름 (자문의견서·민사 서면)

```bash
# 1) 안건명 키워드로 검색 (예: "임차" → 임차인 관련 해석례)
python scripts/law_api.py search --target expc --query "임차" --display 20 --pretty

# 2) 검색 결과의 <법령해석례일련번호>로 본문 조회
python scripts/law_api.py get --target expc --id 332741 --pretty
```

### 인용 형식 (서면용)

법령해석례는 다음 형식으로 인용한다.

- 정식: **법제처 21-0624 회신(2021. 12. 7.)**
- 약식: **법제처 21-0624 해석례**
- 본문 인용 시: 안건명 + `<회답>` 요지를 핵심으로 인용하고, 필요하면 `<이유>`의 결정적 단락을 발췌.

> 법령해석례는 법원의 판례와 달리 **유권해석**으로서 행정청을 사실상 구속하지만, 법원을 구속하지는 않는다.
> 자문의견서에서는 법제처 해석례가 의뢰인 입장과 일치하면 강력한 보강 자료가 되며, 반대 해석이 있다면 그 점도 함께 제시해야 한다.

## 6-C. 과거(연혁) 법령 (eflaw / ordin nw=2 / admrulOldAndNew) — 공식 가이드 + 라이브 검증

### 시행일 법령 (target=eflaw) — 법령의 시점별 버전

검색(`lawSearch.do?target=eflaw`) 주요 파라미터 (공식 가이드 기준):

| 파라미터 | 의미 | 비고 |
|---|---|---|
| `LID` | **법령ID 한정** | 해당 법령의 버전만 반환 — 부분일치 오염(예: "민법" → 난민법) 차단. 권장 |
| `nw` | 1 연혁 / 2 시행예정 / 3 현행 (콤마 조합) | 기본 전체. 본 skill 기본값 `1,3`(시행예정 배제) |
| `sort` | `efasc`/`efdes` 등 | 시행일자 정렬 |
| `efYd` | `YYYYMMDD~YYYYMMDD` | 시행일자 범위 압축 |

응답은 `<LawSearch>`/`<law>`로 현행 검색과 동일 구조이되 **같은 법령이 시행일자별 행으로 반복**된다(버전별 `법령일련번호`=MST, `현행연혁코드`, 공포일자·공포번호 포함). 동일 (시행일자, MST) 중복 행과 **동일 MST가 복수 시행일자로 나오는 경우**(조문별 시행일 차이)가 있으므로 dedupe 후 사용한다.

본문(`lawService.do?target=eflaw`):
- **`MST=` 사용 시 `efYd=`(그 버전의 시행일자)가 필수** (공식 가이드).
- ⚠ **`ID=`(법령ID)를 주면 현행 본문이 반환되고 efYd는 무시**된다 — 과거본 조회에 ID 사용 금지(law_api.py가 차단).
- `JO`(조번호)·`chrClsCd`(010202 한글/010201 원문) 지원. 응답에는 조문시행일자문자열·부칙·별표내용/별표서식(PDF)파일링크·개정문내용·제개정이유내용이 포함되어 연혁 분석에 유용.
- 실측: 옛 MST는 `target=law`로도 본문이 반환되나(폴백 가능) 공식 경로는 eflaw.

버전 선택 규칙(law_api.py `get-asof` 내장): (시행일자, MST) dedupe → **max{시행일자 ≤ 기준일}** → 그 버전의 시행일자를 efYd로 전달. 기준일 이전 버전 부재(제정 전) 시 오류 + 최초 시행일 안내.

> `target=lsHistory`는 존재하지 않는다(라이브 검증 — HTML 오류 페이지).
> 법령 신구법 비교(`target=oldAndNew`, 구조문목록/신조문목록/신구법존재여부)는 공식 가이드가 확보되어 있으나 본 skill에는 미통합(2단계 후보).

### 자치법규 연혁 (target=ordin, nw)

- 검색 파라미터 `nw`: **1 현행(기본) / 2 연혁**. v1.1.0까지의 "자치법규 연혁 미지원" 기재는 nw 미지정 실측에 따른 오류였다.
- 연혁 행의 `자치법규일련번호`로 기존 본문 조회(`lawService.do?target=ordin&MST=`)가 그대로 동작(라이브 검증 — 2006년 시행본 회수).
- `org`(시·도)·`sborg`(시·군·구, org 필수 동반)·`knd`(30001 조례/30002 규칙 등)로 동명 조례를 구분.
- ⚠ 조례는 개정 시 명칭 변경이 잦다 — 정확명 필터로 거르면 옛 버전이 누락되므로 law_api.py는 ordin에 정확명 필터를 걸지 않고 명칭을 그대로 노출한다.

### 행정규칙 연혁 (target=admrul, nw) / 신구법비교 (admrulOldAndNew)

- **행정규칙 목록 조회도 `nw` 파라미터를 지원한다**: 1 현행(기본) / 2 연혁 (공식 가이드. v1.2.0의 "행정규칙은 연혁 검색 API가 없다" 기재는 nw 미지정 실측에 따른 오류 — 자치법규와 동일 패턴이었다). 보조 파라미터: `knd`(1훈령/2예규/3고시/4공고/5지침/6기타), `date`(발령일자), `prmlYd`(발령일자 기간), `modYd`(수정일자 기간), `sort=efasc/efdes`. 응답 항목은 `<admrul>` element이며 `<현행연혁구분>` 필드로 현행/연혁을 구분한다.
- 연혁 행의 `행정규칙일련번호`로 기존 본문 조회(`lawService.do?target=admrul&ID=`)가 그대로 동작한다(라이브 검증 — 현행여부 N 전문 회수). 본문 응답에는 부칙·별표(`별표서식파일링크`·`별표서식PDF파일링크`)·`첨부파일링크` 필드도 정의되어 있다(공식 가이드).
- law_api.py `versions/get-asof --target admrul`은 `--query`(현행+nw=2 병합 검색, **기본·권장**) 또는 `--id`(체인 역추적, 보조)를 받는다.
- 신구법비교(admrulOldAndNew): 목록 `lawSearch.do?...&query=`(현행 단위), 본문 `lawService.do?...&ID=<행정규칙일련번호>` → `<구조문_기본정보>`/`<신조문_기본정보>` + 구조문목록/신조문목록(`신구법존재여부=N`이면 대비 없음).
- **체인 역추적(보조, 라이브 검증)**: `<구조문_기본정보>`의 일련번호(직전 버전)로 ① admrul 과거 전문 조회 ② admrulOldAndNew 재호출이 모두 가능 — 연혁 검색 교차 검증·조문 대비용(기본 상한 15단계, 단계당 API 1회).

## 8. 보안 주의

- OC 값은 **사용자 식별자**다. 코드·로그·서면에 평문으로 노출하지 않는다.
- 환경변수 `LAW_GO_KR_OC`로 관리하고, `--save-to`로 저장한 파일에는 OC가 포함되지 않으므로 그 파일은 공유 가능.
- API URL은 OC를 포함하므로(`?OC=...`), URL 자체를 외부에 공유하지 않는다.
